# 已访问帖子缓存手动测试要点

## 基础点击反馈

1. 打开帖子列表，选择一个标题还是黑色的普通帖子。
2. 点击该帖子进入详情页。
3. 预期：点击后列表对应 row 在跳转前或跳转过程中立即变成灰色标题。
4. 从详情页返回列表。
5. 预期：刚才点击的帖子标题仍然是灰色，不需要手动刷新列表。

## 列表刷新保持

1. 找到一个已经点过、标题已变灰的帖子。
2. 下拉刷新当前列表。
3. 预期：刷新完成后，如果该帖子仍在列表里，标题继续保持灰色。

## 分类与排序保持

1. 点开某个帖子，让它变成灰色。
2. 切换到其他分类，再切回原分类。
3. 预期：该帖子仍然显示为灰色。
4. 切换排序方式。
5. 预期：重新加载出的同一个帖子仍然显示为灰色。

## 重启保持

1. 点开一个帖子，确认列表标题变灰。
2. 让 App 进入后台，等待 1 秒左右。
3. 彻底结束 App 进程后重新打开。
4. 回到列表。
5. 预期：同一个帖子如果出现在列表里，仍然显示灰色。

## 单行更新观察

1. 点击一个未访问帖子。
2. 观察列表跳转前的变化。
3. 预期：只有被点击的 row 变灰，列表整体不应闪烁、不应回到顶部、不应整页 reload。

## 1500 条缓存边界

这个边界正常手动测试成本较高，不建议人工逐条验证。自动化测试已覆盖内存裁剪和 SQLite 裁剪逻辑。人工只需要确认普通访问记录可跨重启保留即可。

## 异常观察

- 如果点击后没有立即变灰，优先检查是否点中了非帖子入口或列表是否已经刷新到其他分类。
- 如果重启后变灰状态丢失，重点看 App 是否有进入后台触发落盘，或控制台是否有 SQLite 打开/写入错误。
